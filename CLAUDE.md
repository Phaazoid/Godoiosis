# Iosis ‚Äî Project Guide

Tactical RPG (Fire Emblem-influenced), Godot 4.6, GDScript. Solo hobbyist developer.

## Collaboration contract (read first)

- **The user types all gameplay code themselves** (`Classes/`, `Scenes/`, `game.gd`). Guide with complete code blocks and file/line anchors; never edit those files directly unless explicitly asked. The user is here to learn ‚Äî explain the *why*, not just the *what*.
- **Deliver every change as a typed code block.** Prose-described steps ("then update the click handler to...") reliably fail to get implemented. If the user must type it, write it out fully.
- Claude MAY directly create/edit: `CLAUDE.md`, `docs/`, `tests/`, GitHub issue text, and other non-gameplay scaffolding (standing exception granted 2026-06-12).
- After walkthroughs land, *verify by reading the actual files* before debugging from theory ‚Äî transcription drift is the most common failure mode.

Run **`/agent-queue`** to have Claude scan the `agent/claude` issues and advance each one, then hand back. ## Design laws (non-negotiable)

1. **No randomness in gameplay.** No hit/miss rolls, no crit chance, nothing. Combat is fully deterministic. (Old design docs mentioning crits predate this law.)
2. **The action queue never lies.** Planned actions must preview exactly what execution does. Queue/cancel/requeue must be idempotent. Derived actions (counter-attacks) are *computed* from the plan, never stored as player orders.
3. **Future AI uses the player's API.** AI issues orders exclusively through `SquadManager.queue_action` ‚Äî no side channels. This protects law #2 forever.

## Architecture map

- `game.gd` ‚Äî input/game-state coordinator. Known-overweight; prefer moving domain logic out when touching it.
- `SquadManager` ‚Äî **the only owner of squad lifecycle**: create/destroy/join/leave. Member removal funnels through `Squad._erase_member()` (the sole `members.erase` caller, mirroring `_add_member`); its callers are `_detach_from_current_squad()` (single-unit) and `disband_squad()` (bulk). Also home of counter-attack resolution.
- `Squad` ‚Äî member list + `action_queue` (player-authored orders only). Enforces one order per action-type per unit (volley siblings exempt ‚Äî see `_is_volley_sibling`).
- **Persistence seam (locked, #8):** transient `Unit` node holds battle-scoped state that resets each mission (element states, projected position, lifecycle ACTIVE/DOWNED/DEAD); persistent `UnitInstance` resource holds identity that survives missions (HP, fixed stats, limb loss, proficiency, **Will**). New per-unit state picks its side by one test ‚Äî *does it survive a mission?* No third store. (Cross-mission save/load is the future campaign layer; boundary is locked now so features land on the right side. See `docs/design/resolution-pipeline.md` ‚Üí "The persistence seam".)
- Actions: `BaseAction` ‚Üí `MoveAction`, `AttackAction` (‚Üí `CounterAttackAction`), `RescueAction` (#33). **Main actions** (`BaseAction.is_main_action()` ‚Äî attack + rescue today) are mutually exclusive and must come *after* any move: one per unit per turn, and locking one removes the unit's move option (enforced at the menu and in `SquadManager.queue_action`). AoE = a **volley**: one `AttackAction` per victim, all sharing a `volley` array; secondaries set `is_secondary_hit` and skip the lunge animation.
- `CombatComponent` ‚Äî weapon-aware reach: `get_attack_cells_from` (facing-specific), `get_all_attack_cells_from` (union over 4 facings, for overlays), `get_affected_cells_from` (AoE), `is_directional_attack()` (does the equipped pattern aim by facing?).
- **Targeting modes (#25):** patterns are **directional** (`AttackPattern.is_directional()` true ‚Äî forward line/wide: the player points a cardinal direction and the whole spread fires; the pointed cell need not be a spread member) or **point** (false ‚Äî Manhattan / no weapon: the hovered cell itself must be in range). `game.gd` ATTACK_TARGETING branches on `CombatComponent.is_directional_attack()` for both hover-preview and click; the red overlay still shows the full 4-facing reach (`get_all_attack_cells_from`). ForwardWide `width` is **odd-only** (`@export_enum`); `ManhattanRangePattern` supports a no-float blended Manhattan/Chebyshev range (`max_and_a_half` bevels in the diagonal corners of the max ring ‚Üí `GridUtils.cells_within_blended_range`). The attacker's chosen facing is now explicit ‚Äî the intended seam for a future **backstrike** (facing-vs-facing damage).
- `WeaponData` = **policy** (power, scaling_stat, can_counter, hits_allies) + `AttackPattern` resources = **geometry** (selectable cells vs affected cells). Patterns stay pure geometry; terrain/LoS filtering belongs at the resolution layer when it arrives (planned weapon flag: `arcs_over_obstacles`).
- Execution order: moves (parallel) ‚Üí attacks (sequential ‚Äî combo order matters for future elemental) ‚Üí counters (sequential) ‚Üí rescues (sequential). Units downed mid-pass are queued and ejected from their squads **after** the whole pass (`_process_downed_pending`), never mid-`execute` (restructuring squads mid-resolution was buggy).
- `TurnManager` (`flow/`) ‚Äî the faction turn cycle, **rebuilt from the board each hand-off** (refactored 2026-06-24). Tracks the **active faction by value** (the old two-sided `TurnPhase` enum is gone); `end_turn(present_factions)` recomputes the live order as `Team.all_factions()` (enum order) ‚à© factions present, then advances past the active one. Handles **any number of teams**, and the cycle **builds itself from what's on the map** ‚Äî a newly-spawned faction enrolls at the next hand-off, a wiped one drops out. **Single source of truth = the `Team.Faction` enum** (add a member ‚Üí it's in the rotation). Trade-off: turn order follows enum-declaration order, which is **append-only** for save-compat, so a new faction acts *last* unless you migrate enum values; and presence *is* eligibility (a `NEUTRAL`/`OTHER` faction with units would take a turn ‚Äî add an explicit opt-out if you ever want non-acting scenery). `game._on_turn_started(faction)` **auto-skips** a faction with no `is_active` units after ticking its downed clocks ‚Äî guarded by `_board_has_active_units()` so an all-downed board can't infinite-loop; `start_faction_turn(faction)` replaced the old player/enemy split. Saved turn state = `ScenarioData.active_faction` (authoritative; the `turn_phase` field was removed), restored via `set_active_faction`. The headless Play API mirrors this exactly (`play_session.end_turn` derives present factions + runs the same guarded skip).
- **Archetype AI (`ai/`, #29, built 2026-07-01):** `AIController` (child of Game) runs AI-controlled factions ‚Äî per squad: fixed-duration camera `pan_to` the leader ‚Üí archetype plans ‚Üí the same `game.execute_orders` the player's Execute button calls. Orders go through `SquadManager.queue_action`/`queue_group_move` ONLY (Law #3). Two independent layers: **ENABLED** is per-faction, session-only (Scenario-tab checkboxes, default OFF ‚Äî a kill switch back to manual/hotseat control); **ARCHETYPE** is per-squad and persists with the scenario (`Squad.archetype`, an `AIArchetype.Type` ‚Äî saved with `squad_name`/`squad_zone` on the leader's `ScenarioUnitEntry`; the sentry post = the leader's saved `cell`). Archetypes (`AIArchetype` enum+registry, `AITactics` shared queries): RUSHDOWN (nearest enemy ‚Üí path ‚Üí attack), HOLD (attack in reach, never move), SENTRY (idle at post; enemy inside its painted zone ‚Üí engage with every destination clamped to zone‚à™post ‚Äî `plan_group_move(allowed_cells)` leashes members too; zone clear ‚Üí walk back to post; zone missing ‚Üí Hold). **Downed enemies are deprioritized, not protected (#57, 2026-07-15):** `AITactics.nearest_enemy`/`_nearest_reachable_enemy` both prefer any ACTIVE enemy and fall through to a downed one only when nothing active qualifies (fork 3 ‚Äî finishing a downed unit is legal, just not preferred) ‚Äî a preference, not a ban. Zones: `ZoneManager` (name‚Üícells, game child), painted via the Tile Brush tab's Zone mode, persisted in `ScenarioData.zones`, drawn on `ZoneOverlay` ONLY while the Tile Brush tab is up. During AI turns `GameState.AI_TURN` locks all board/queue-panel input and the camera. **AI Crisis stances BUILT (#57, 2026-07-15):** `_offer_crisis` no longer blanket-auto-accepts ‚Äî every non-PLAYER faction (AI-controlled or hotseat) decides via `AIArchetype.accepts_crisis(archetype)`, a deterministic per-archetype table (`AIArchetype.CRISIS_STANCES`: RUSHDOWN always, HOLD/SENTRY never) declared on the registry; the PLAYER faction alone keeps the live `CrisisPrompt`. `ResolvedOutcome.Lethality.CRISIS` threads through `PlanResolver` so the queue previews it exactly (R9: enemy Crisis is never a BREAK) ‚Äî `PlanResolver._predict_lethality` now takes the threaded `_Hypo` whole (was 8 positional params, a call-site hazard). Tests: `tests/ai/` (40 cases).
- `OverlayManager` ‚Äî all board visuals (tile overlays, icons, path arrows, projected units).
- Dev tools: `Classes/dev/DevOverlay.gd` (+ per-tool scripts in `Classes/dev/`) ‚Äî spawner, reflection-based weapon editor (`get_property_list` walker), pattern swapper (`ProjectSettings.get_global_class_list`), sprite catalog (folder scan).
- Registries: `Classes/core/Stats.gd` (`STAT_DEFAULTS` is the canonical stat list ‚Äî input stats STR/DEX/PER/CON; missing dict keys read defaults, never 0; also home of the stat-shadow doctrine: `*_band` helpers + `armor_def`, pure statics per `docs/design/stats.md`), `Classes/weapons/WeaponCatalog.gd`. Rule: game content lists ‚Üí domain-named registry; single-system config ‚Üí stays local.

## Sharp edges (each of these has already bitten once)

- **Always `duplicate(true)`** when granting weapons/resources to units ‚Äî shallow `duplicate()` shares the nested `attack_pattern`.
- `.tres` files omit properties at default values ‚Äî an "empty-looking" resource may just be unsaved-at-defaults.
- UI rule: **scene for static, code for dynamic.** Fixed controls live in `.tscn`; data-shaped UI (editor fields, stat rows) is generated.
- Plain `Panel`s report 0√ó0 minimum size for absolutely-positioned children inside containers.
- `Unit.has_squad()` means "has squad*mates*" (members > 1). Every unit always belongs to exactly one managed squad, even solo.
- Godot ternary exists (`x if cond else y`); typed dictionaries (`Dictionary[String, int]`) are used throughout ‚Äî keep chains typed end-to-end.
- **Retyping a persisted `@export` field from String to an enum is a *data* migration, not just a code change.** Godot silently *drops* type-mismatched values when loading a `.tres` (a saved `Dictionary[String, int]` won't populate a `Dictionary[Stats.Stat, int]` ‚Äî it loads empty) and *strips* them on the next resave. After retyping such a field (stat-dict keys, `weapon_type`, `scaling_stat`), rewrite every saved `.tres` to the enum form ‚Äî enums serialize as plain ints (`base_stats = Dictionary[int, int]({0: 20, ‚Ä¶})`, `weapon_type = 1`). Bit 2026-06-19 (#7 fallout): every unit loaded with 0 stats ‚Üí LDR 0 ‚Üí nobody could move; the only intact copy of the string data was git HEAD, since the migration commit never touched the `.tres`. Migrate from HEAD, not the working tree (Godot may have already stripped it).
- **Dev tools = separate OS window; the game is wrapped in a SubViewport (June 2026 refactor).** Scene tree (`Main.tscn` is the main scene): `Main` (Node) ‚Üí `GameContainer` (SubViewportContainer, Stretch on, texture_filter Nearest) ‚Üí `GameView` (SubViewport, embed ON) ‚Üí `Game` (game.tscn instance); `DevOverlay` (a `Window`, embed ON) is a sibling of `GameContainer` under `Main`. Project `embed_subwindows` = **OFF** ‚Üí `DevOverlay` is a real OS window (second-monitor capable); `GameView` embeds so the game's own popups (action menu) stay in-game. Cross-boundary refs: `game.gd` ‚Üí `get_node("/root/Main/DevOverlay")`; `dev_overlay.gd` ‚Üí `get_node("../GameContainer/GameView/Game")` (+ `‚Ä¶/Game/ScenarioManager`). Gotchas this created ‚Äî all already handled, **do not undo**: (1) SubViewports default to LINEAR filtering; `game._ready` sets the viewport default to Nearest via `RenderingServer.viewport_set_default_canvas_item_texture_filter(get_viewport().get_viewport_rid(), RenderingServer.CANVAS_ITEM_TEXTURE_FILTER_NEAREST)` ‚Äî per-node `texture_filter` only patches part of the tree (the chain breaks at any `CanvasLayer` / non-CanvasItem ancestor). (2) Embedded popups don't auto-dismiss on outside-click (input is forwarded through the container); `ActionMenuController._input` closes them manually. (3) Stretch mode = `disabled`; UI is real pixels ‚Äî no hardcoded pixel positions (use viewport-relative; the hover panel's old `BOTTOM_LEFT_POS=410` broke on a viewport-height change). (4) Any new subwindow UI (PopupMenu/dialogs) inherits these quirks ‚Äî prefer Control-based menus.
