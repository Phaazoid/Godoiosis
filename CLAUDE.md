# Iosis — Project Guide

Tactical RPG (Fire Emblem-influenced), Godot 4.6, GDScript. Solo hobbyist developer.

## Collaboration contract (read first)

- **The user types all gameplay code themselves** (`Classes/`, `Scenes/`, `game.gd`). Guide with complete code blocks and file/line anchors; never edit those files directly unless explicitly asked. The user is here to learn — explain the *why*, not just the *what*.
- **Deliver every change as a typed code block.** Prose-described steps ("then update the click handler to...") reliably fail to get implemented. If the user must type it, write it out fully.
- Claude MAY directly create/edit: `CLAUDE.md`, `docs/`, `tests/`, `play/` (the headless Play API — confirmed 2026-07-16 as already Claude's domain, not user-typed gameplay code), GitHub issue text, and other non-gameplay scaffolding (standing exception granted 2026-06-12).
- **Standing exception (2026-07-16):** when a walkthrough touches a `Classes/`/`Scenes/`/`game.gd` file that has no top-of-file description comment (what the file/class IS and its place in the architecture), Claude adds that comment directly instead of asking the user to type it. Everything else in the file stays hand-typed. Start-of-session sweep target: any file touched by the issue in progress.
- **Standing exception (2026-07-16): `.tscn` scenes and saved scenarios.** If a code change breaks an existing `.tscn` file or a saved scenario `.tres` (`Scenarios/*.tres`), Claude has full permission to edit it directly to keep it working — current scene/scenario content is disposable placeholder, and a clean codebase (no legacy-compat branches carried for their sake) matters more than preserving them untouched. Doesn't extend to gameplay `.gd` code in `Classes/`/`game.gd`, which stays hand-typed regardless.
- **Standing exception (2026-07-16): comments in `Classes/`/`game.gd`.** Claude may edit comments directly (stale references, renamed symbols, drifted explanations) even in files whose code stays hand-typed — comments aren't code. Actual logic in these files is still the user's to type.
- After walkthroughs land, *verify by reading the actual files* before debugging from theory — transcription drift is the most common failure mode.

## GitHub issue workflow (Claude ↔ human handoff)

Open work lives in [GitHub Issues](https://github.com/Phaazoid/Godoiosis/issues). Two **mutually exclusive** assignment labels say whose turn it is (exactly one per open issue):

- **`agent/claude`** — Claude owns the next step: draft/revise a fix walkthrough, or (for `tests/` & `docs/`) just do the work directly.
- **`agent/human`** — a human owns the next step: type a posted walkthrough, make a design decision, or test. Flip an issue back to `agent/claude` (with a reply) when a fix needs rework.

Run **`/agent-queue`** to have Claude scan the `agent/claude` issues and advance each one, then hand back. Every comment Claude posts under the user's account **leads with `🤖 Claude says:`** and **ends with** `— Claude (<model>) · <date>` (name the model actually running, e.g. Fable 5 / Opus 4.8 — the signature is provenance), authored via the Write tool → `gh issue comment --body-file` (never an inline non-ASCII arg — Windows PowerShell 5.1 mojibakes it before upload; see the encoding note in `tests/README.md`). After acting, Claude flips the label to `agent/human`.

**Canon-sweep discipline before calling a doc update done (added 2026-07-16, after a full-corpus staleness sweep found 9/22 `docs/design/` files drifted):** a "canon swept" claim means a grep-verified pass over the *whole* `docs/design/` tree (+ this file) for the issue's own number and any renamed/removed code symbol or term it touched — not just an edit to the one file that's obviously the topic's home. Staleness hides in three places a narrow, feature-specific edit misses: (1) a status banner elsewhere in the *same* file that a detail-section-only edit didn't reach; (2) a sibling file stating the same fact from a different angle (weapons.md and elemental-system.md both independently described issue #7 as open after it closed); (3) a "living list" tracker (`docs/design/grill-queue.md`) that can go stale from a *different* ticket's work than the one closing. For multi-session issues, sweep against the issue's *final* state at close — re-read its own comment history, not just what the current session built.

**Staleness marker (added 2026-07-16):** every living design doc (all of `docs/design/` except the three deliberately-frozen historical records — the coherence audit, the co-dev agenda, the wiki triage ledger) carries a `**Canon checked through #NN (date).**` line in its opening banner. When a sweep re-verifies a doc, bump its marker to the current highest issue number + that day's date. A marker sitting far behind the current issue count is the at-a-glance signal a doc is due for a re-check — don't let it go stale by forgetting the bump once you've actually re-verified the doc.

## Design laws (non-negotiable)

1. **No randomness in gameplay.** No hit/miss rolls, no crit chance, nothing. Combat is fully deterministic. (Old design docs mentioning crits predate this law.)
2. **The action queue never lies.** Planned actions must preview exactly what execution does. Queue/cancel/requeue must be idempotent. Derived actions (counter-attacks) are *computed* from the plan, never stored as player orders.
3. **Future AI uses the player's API.** AI issues orders exclusively through `SquadManager.queue_action` — no side channels. This protects law #2 forever.

## Architecture map

- `game.gd` — input/game-state coordinator. Known-overweight; prefer moving domain logic out when touching it.
- `SquadManager` — **the only owner of squad lifecycle**: create/destroy/join/leave. Member removal funnels through `Squad._erase_member()` (the sole `members.erase` caller, mirroring `_add_member`); its callers are `_detach_from_current_squad()` (single-unit) and `disband_squad()` (bulk). Also home of counter-attack resolution.
- `Squad` — member list + `action_queue` (player-authored orders only). Enforces one order per action-type per unit (volley siblings exempt — see `_is_volley_sibling`).
- **Persistence seam (locked, #8):** transient `Unit` node holds battle-scoped state that resets each mission (element states, projected position, lifecycle ACTIVE/DOWNED/DEAD); persistent `UnitInstance` resource holds identity that survives missions (HP, fixed stats, limb loss, proficiency, **Will**). New per-unit state picks its side by one test — *does it survive a mission?* No third store. (Cross-mission save/load is the future campaign layer; boundary is locked now so features land on the right side. See `docs/design/resolution-pipeline.md` → "The persistence seam".)
- Actions: `BaseAction` → `MoveAction`, `AttackAction` (→ `CounterAttackAction`), `RescueAction` (#33). **Main actions** (`BaseAction.is_main_action()` — attack + rescue today) are mutually exclusive and must come *after* any move: one per unit per turn, and locking one removes the unit's move option (enforced at the menu and in `SquadManager.queue_action`). AoE = a **volley**: one `AttackAction` per victim, all sharing a `volley` array; secondaries set `is_secondary_hit` and skip the lunge animation.
- `CombatComponent` — weapon-aware reach: `get_attack_cells_from` (facing-specific), `get_all_attack_cells_from` (union over 4 facings, for overlays), `get_affected_cells_from` (AoE), `is_directional_attack()` (does the equipped pattern aim by facing?).
- **Targeting modes (#25):** patterns are **directional** (`AttackPattern.is_directional()` true — forward line/wide: the player points a cardinal direction and the whole spread fires; the pointed cell need not be a spread member) or **point** (false — Manhattan / no weapon: the hovered cell itself must be in range). `game.gd` ATTACK_TARGETING branches on `CombatComponent.is_directional_attack()` for both hover-preview and click; the red overlay still shows the full 4-facing reach (`get_all_attack_cells_from`). ForwardWide `width` is **odd-only** (`@export_enum`); `ManhattanRangePattern` supports a no-float blended Manhattan/Chebyshev range (`max_and_a_half` bevels in the diagonal corners of the max ring → `GridUtils.cells_within_blended_range`). The attacker's chosen facing is now explicit — the intended seam for a future **backstrike** (facing-vs-facing damage).
- **Weapons are template + instance, and the attack is its own content (#59 then #72, mirrors the jobs id-reference pattern + the rune/carving system):** `AttackData` (`extends Resource`) is the shared skeleton anything a unit can fire carries — `display_name`, `power`, `attack_pattern`, `can_counter`, `hits_allies`, `targets` — with damage math deliberately OFF the base (carvings scale off aura, weapon attacks off the wielding weapon's blend + mods). `WeaponAttackData extends AttackData` adds only `elemental_damage_type`; `TransmutationData` (the rune carving) is its sibling subclass. `WeaponData` is the shared family TEMPLATE (family base like `ChainSword.tres`, or a named prototype) — `main_attack: WeaponAttackData` (the one REQUIRED attack, a curated resource the family points at — never a bare array slot — used by default aim and ALWAYS by counters) + `extra_attacks: Array[WeaponAttackData]` (no count cap; mod spaces gate access), plus `scaling_blend` (a `Dictionary[Stats.Stat, int]` % blend across STR/DEX/PER/CON), `weapon_type`, `base_weight`, `is_prototype`. It is NOT equippable (extends `Item`, not `EquippableData`) and is never duplicated — a unit that owns a template edits it live for every weapon built on it, main attack included. `WeaponInstance` (extends `EquippableData`, what a unit actually carries) holds a direct `template: WeaponData` ref plus its own `space_1`/`space_2`/`space_3` (fitted `WeaponModData`, the only true per-weapon state) and computes the effective view (`base_damage`/`get_elements`/`hits_map`, all taking an optional `attack: WeaponAttackData` override that defaults to main when null) from template + active modules. **Never `duplicate(true)` a `WeaponInstance`** — that deep-copies the shared template and severs the sync; use `copy_equippable()` (the `EquippableData` virtual every equippable overrides as needed — `WeaponInstance`'s keeps `template` shared and copies spaces shallowly, the default falls back to `duplicate(true)` for `RuneData`). In battle, `Unit.get_fired_attack()` (live pick else first-channelable-carving/main) drives reach + ally-splash; `Unit.get_counter_attack()` is a deliberate exception — weapons always counter with main, ignoring any live pick (a rune counters with whatever it fires; overwatch-style alt-attack countering is future scope, #73) — and the chosen `AttackData` is stamped onto `AttackAction.fired_attack` at declare time so the resolver reads a frozen snapshot, never re-derives (Law #2). `AttackPattern` resources stay pure **geometry** (selectable cells vs affected cells), unaffected by any of this split; terrain/LoS filtering belongs at the resolution layer when it arrives (planned weapon flag: `arcs_over_obstacles`).
- Execution order: moves (parallel) → attacks (sequential — combo order matters for future elemental) → counters (sequential) → rescues (sequential). Units downed mid-pass are queued and ejected from their squads **after** the whole pass (`_process_downed_pending`), never mid-`execute` (restructuring squads mid-resolution was buggy).
- `TurnManager` (`flow/`) — the faction turn cycle, **rebuilt from the board each hand-off** (refactored 2026-06-24). Tracks the **active faction by value** (the old two-sided `TurnPhase` enum is gone); `end_turn(present_factions)` recomputes the live order as `Team.all_factions()` (enum order) ∩ factions present, then advances past the active one. Handles **any number of teams**, and the cycle **builds itself from what's on the map** — a newly-spawned faction enrolls at the next hand-off, a wiped one drops out. **Single source of truth = the `Team.Faction` enum** (add a member → it's in the rotation). Trade-off: turn order follows enum-declaration order, which is **append-only** for save-compat, so a new faction acts *last* unless you migrate enum values; and presence *is* eligibility (a `NEUTRAL`/`OTHER` faction with units would take a turn — add an explicit opt-out if you ever want non-acting scenery). `game._on_turn_started(faction)` **auto-skips** a faction with no `is_active` units after ticking its downed clocks — guarded by `_board_has_active_units()` so an all-downed board can't infinite-loop; `start_faction_turn(faction)` replaced the old player/enemy split. Saved turn state = `ScenarioData.active_faction` (authoritative; the `turn_phase` field was removed), restored via `set_active_faction`. The headless Play API mirrors this exactly (`play_session.end_turn` derives present factions + runs the same guarded skip).
- **Archetype AI (`ai/`, #29, built 2026-07-01):** `AIController` (child of Game) runs AI-controlled factions — per squad: fixed-duration camera `pan_to` the leader → archetype plans → the same `game.execute_orders` the player's Execute button calls. Orders go through `SquadManager.queue_action`/`queue_group_move` ONLY (Law #3). Two independent layers: **ENABLED** is per-faction, session-only (Scenario-tab checkboxes, default OFF — a kill switch back to manual/hotseat control); **ARCHETYPE** is per-squad and persists with the scenario (`Squad.archetype`, an `AIArchetype.Type` — saved with `squad_name`/`squad_zone` on the leader's `ScenarioUnitEntry`; the sentry post = the leader's saved `cell`). Archetypes (`AIArchetype` enum+registry, `AITactics` shared queries): RUSHDOWN (nearest enemy → path → attack), HOLD (attack in reach, never move), SENTRY (idle at post; enemy inside its painted zone → engage with every destination clamped to zone∪post — `plan_group_move(allowed_cells)` leashes members too; zone clear → walk back to post; zone missing → Hold). **Downed enemies are deprioritized, not protected (#57, 2026-07-15):** `AITactics.nearest_enemy`/`_nearest_reachable_enemy` both prefer any ACTIVE enemy and fall through to a downed one only when nothing active qualifies (fork 3 — finishing a downed unit is legal, just not preferred) — a preference, not a ban. Zones: `ZoneManager` (name→cells, game child), painted via the Tile Brush tab's Zone mode, persisted in `ScenarioData.zones`, drawn on `ZoneOverlay` ONLY while the Tile Brush tab is up. During AI turns `GameState.AI_TURN` locks all board/queue-panel input and the camera. **AI Crisis stances BUILT (#57, 2026-07-15):** `_offer_crisis` no longer blanket-auto-accepts — every non-PLAYER faction (AI-controlled or hotseat) decides via `AIArchetype.accepts_crisis(archetype)`, a deterministic per-archetype table (`AIArchetype.CRISIS_STANCES`: RUSHDOWN always, HOLD/SENTRY never) declared on the registry; the PLAYER faction alone keeps the live `CrisisPrompt`. `ResolvedOutcome.Lethality.CRISIS` threads through `PlanResolver` so the queue previews it exactly (R9: enemy Crisis is never a BREAK) — `PlanResolver._predict_lethality` now takes the threaded `_Hypo` whole (was 8 positional params, a call-site hazard). Tests: `tests/ai/` (40 cases).
- `OverlayManager` — all board visuals (tile overlays, icons, path arrows, projected units).
- Dev tools: `Classes/dev/DevOverlay.gd` (+ per-tool scripts in `Classes/dev/`) — spawner, reflection-based weapon editor (`get_property_list` walker), pattern swapper (`ProjectSettings.get_global_class_list`), sprite catalog (folder scan).
- Registries: `Classes/core/Stats.gd` (`STAT_DEFAULTS` is the canonical stat list — input stats STR/DEX/PER/CON; missing dict keys read defaults, never 0; also home of the stat-shadow doctrine: `*_band` helpers + `armor_def`, pure statics per `docs/design/stats.md`), `Classes/weapons/WeaponCatalog.gd`, `Classes/jobs/JobCatalog.gd`. Rule: game content lists → domain-named registry; single-system config → stays local.

## Sharp edges (each of these has already bitten once)

- **Grant/copy an equippable via `copy_equippable()`, never a bare `duplicate(true)`** (revised #59 — this used to say the opposite). `RuneData` still wants a full deep copy (its base implementation), but `WeaponInstance` overrides it to keep `template` a SHARED reference while its own `space_1/2/3` copy shallowly — a generic `duplicate(true)` would deep-copy the template too and silently fork a weapon off its family.
- `.tres` files omit properties at default values — an "empty-looking" resource may just be unsaved-at-defaults.
- UI rule: **scene for static, code for dynamic.** Fixed controls live in `.tscn`; data-shaped UI (editor fields, stat rows) is generated.
- Plain `Panel`s report 0×0 minimum size for absolutely-positioned children inside containers.
- `Unit.has_squad()` means "has squad*mates*" (members > 1). Every unit always belongs to exactly one managed squad, even solo.
- Godot ternary exists (`x if cond else y`); typed dictionaries (`Dictionary[String, int]`) are used throughout — keep chains typed end-to-end.
- **Retyping a persisted `@export` field from String to an enum is a *data* migration, not just a code change.** Godot silently *drops* type-mismatched values when loading a `.tres` (a saved `Dictionary[String, int]` won't populate a `Dictionary[Stats.Stat, int]` — it loads empty) and *strips* them on the next resave. After retyping such a field (stat-dict keys, `weapon_type`, `scaling_stat`), rewrite every saved `.tres` to the enum form — enums serialize as plain ints (`base_stats = Dictionary[int, int]({0: 20, …})`, `weapon_type = 1`). Bit 2026-06-19 (#7 fallout): every unit loaded with 0 stats → LDR 0 → nobody could move; the only intact copy of the string data was git HEAD, since the migration commit never touched the `.tres`. Migrate from HEAD, not the working tree (Godot may have already stripped it).
- **Dev tools = separate OS window; the game is wrapped in a SubViewport (June 2026 refactor).** Scene tree (`Main.tscn` is the main scene): `Main` (Node) → `GameContainer` (SubViewportContainer, Stretch on, texture_filter Nearest) → `GameView` (SubViewport, embed ON) → `Game` (game.tscn instance); `DevOverlay` (a `Window`, embed ON) is a sibling of `GameContainer` under `Main`. Project `embed_subwindows` = **OFF** → `DevOverlay` is a real OS window (second-monitor capable); `GameView` embeds so the game's own popups (action menu) stay in-game. Cross-boundary refs: `game.gd` → `get_node("/root/Main/DevOverlay")`; `dev_overlay.gd` → `get_node("../GameContainer/GameView/Game")` (+ `…/Game/ScenarioManager`). Gotchas this created — all already handled, **do not undo**: (1) SubViewports default to LINEAR filtering; `game._ready` sets the viewport default to Nearest via `RenderingServer.viewport_set_default_canvas_item_texture_filter(get_viewport().get_viewport_rid(), RenderingServer.CANVAS_ITEM_TEXTURE_FILTER_NEAREST)` — per-node `texture_filter` only patches part of the tree (the chain breaks at any `CanvasLayer` / non-CanvasItem ancestor). (2) Embedded popups don't auto-dismiss on outside-click (input is forwarded through the container); `ActionMenuController._input` closes them manually. (3) Stretch mode = `disabled`; UI is real pixels — no hardcoded pixel positions (use viewport-relative; the hover panel's old `BOTTOM_LEFT_POS=410` broke on a viewport-height change). (4) Any new subwindow UI (PopupMenu/dialogs) inherits these quirks — prefer Control-based menus.
